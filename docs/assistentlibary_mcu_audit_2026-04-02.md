## 1) Как библиотека работает сейчас

### Архитектура

- `AssistentLibrary` — сетевой слой устройства (Wi-Fi, HTTP endpoint'ы, dispatch команд, отправка телеметрии).
- `AssistentAES` — шифрование/дешифрование сообщений AES + Base64.
- `AssistentVariable` — контейнер переменных устройства и сериализация в JSON-подобную строку.
- `Config` — чтение/запись JSON-конфига в LittleFS.

### Поток запуска

1. В `Begin(...)`:
- инициализируется EEPROM;
- при наличии `cfg` поднимается LittleFS и загружается конфиг;
- устанавливается AES-ключ;
- поднимается Wi‑Fi и HTTP-сервер.

Ключевые места:
- `AssistentLibrary.cpp:272` (`Begin(...)`)
- бесконечное ожидание Wi-Fi: `AssistentLibrary.cpp:314`
- регистрация endpoint'ов:
  - `/command` — `AssistentLibrary.cpp:332`
  - `/clear` — `AssistentLibrary.cpp:333-334`
  - `/ota` — `AssistentLibrary.cpp:336-337`

### Обработка команд

1. Устройство принимает POST `/command` в `Reader()`.
2. Достаёт body (`plain`) и расшифровывает JSON-пакет через `decryptMessageFromJSON(...)`.
3. В стандартном режиме проверяет IP отправителя:
- первый валидный IP «привязывается» в EEPROM;
- дальше принимаются команды только с этого IP.
4. Выполняет встроенные команды (`SERV_GAI`, `SERV_GCFG`, `SERV_SCFG`) или пользовательские команды из массивов `CMD`/`CMDRec`.
5. Шифрует ответ и возвращает JSON.

Ключевые места:
- `Reader()`: `AssistentLibrary.cpp:397`
- стандартный обработчик: `AssistentLibrary.cpp:65`
- проверка IP через `remoteIP()`: `AssistentLibrary.cpp:72`
- встроенные команды: `AssistentLibrary.cpp:227`

### Отправка телеметрии

- `IoTMessage(...)` шифрует payload и отправляет POST на `http://<server_ip>/iot/receive`.
- `server_ip` берётся из RAM/EEPROM.

Ключевые места:
- `AssistentLibrary.cpp:343`
- транспорт по HTTP (без TLS): `AssistentLibrary.cpp:348`

## 2) Основные риски и уязвимости

## Критично

1. Неаутентифицированные служебные endpoint'ы
- `/clear` и `/ota` вызываются без проверки отправителя/подписи запроса.
- Любой клиент в сети может сбросить привязку/конфиг или попытаться запустить OTA.
- Доказательство: `AssistentLibrary.cpp:333-337`, `AssistentLibrary.cpp:58`, `AssistentLibrary.cpp:115`, `AssistentLibrary.cpp:183`.

2. Статический IV + нет контроля целостности
- `aesIv` фиксированный (константа), повторяется между сообщениями.
- Нет HMAC/AEAD, значит возможна модификация ciphertext (malleability) и replay.
- Доказательство: `AssistentAES.h:13`, `AssistentAES.cpp:48`, `AssistentAES.cpp:61`.

3. OTA без криптографической верификации прошивки
- Используется URL из запроса, подпись бинарника не проверяется.
- Для ESP32/ESP8266 при компромете сети это путь к RCE на устройстве.
- Доказательство: `AssistentLibrary.cpp:117`, `AssistentLibrary.cpp:121`, `AssistentLibrary.cpp:188`.

## Высокий приоритет

4. Шифрованный канал поверх HTTP (без TLS)
- Трафик идёт на `http://`, MITM может вмешиваться в транспорт и метаданные.
- Доказательство: `AssistentLibrary.cpp:348`.

5. Ошибки парсинга JSON не обрабатываются
- `deserializeJson(doc, jsonMessage)` вызывается без проверки `error`.
- При битом/пустом JSON возможны `nullptr` в `encryptedMessage/encryptedArg` и краши/DoS.
- Доказательство: `AssistentLibrary.cpp:435-437`, `AssistentLibrary.cpp:445`, `AssistentLibrary.cpp:448`.

6. Риск переполнения стека на данных от сети
- В `decryptMessage` используются VLA с размером от входной строки (`decodeLength`).
- На ESP это может вызвать reset при больших payload.
- Доказательство: `AssistentAES.cpp:30-33`.

7. Неограниченное ожидание Wi‑Fi
- `while (WiFi.status() != WL_CONNECTED)` без timeout/retry стратегии.
- Устройство может зависнуть на старте.
- Доказательство: `AssistentLibrary.cpp:314-319`.

8. Уязвимая схема доверия по IP
- Привязка к первому IP без криптографического подтверждения сервера.
- Первый клиент может «захватить» устройство до легитимного сервера.
- Доказательство: `AssistentLibrary.cpp:72-84`.

9. Debug включён по умолчанию и выводит чувствительные данные
- Макрос `ASSISTENT_DEBUG` жёстко включён в заголовке.
- В лог выводятся encrypted/decrypted поля.
- Доказательство: `AssistentLibrary.h:5`, `AssistentLibrary.cpp:442-451`.

## Средний приоритет

10. Потенциальный null-deref в `ToType`
- `ToType` не проверяет `d == nullptr` перед `d->GetType()`.
- При неизвестном ключе это падение.
- Доказательство: `AssistentVariable.h:21-24`.

11. Ненадёжная ручная сборка JSON строками
- Значения не экранируются (кавычки/спецсимволы), возможен невалидный JSON.
- Доказательство: `AssistentVariable.h:26+`, `DataVariable.h`, `DataList.h`, `DataTemperature.h`.

12. Сигнал об ошибке конфига теряется
- `SERV_SCFG` всегда отвечает `Succesful`, даже если `Inject` не смог распарсить JSON.
- Доказательство: `AssistentLibrary.cpp:259-263`, `Config.h:133-140`.

13. Утечки памяти в жизненном цикле переменных
- Типы создаются через `new`, явного освобождения контейнером нет.
- Для долгоживущих устройств с динамическими добавлениями это накопление утечек.

## 3) Что улучшить (практический план)

### Этап A (сначала закрыть критичные дыры)

1. Закрыть `/clear` и `/ota` аутентификацией
- Минимум: HMAC-SHA256 подпись + timestamp + nonce.
- Лучше: mTLS/HTTPS + серверная авторизация + allowlist адресов.

2. Перейти на AEAD
- Заменить текущую схему на AES-GCM или ChaCha20-Poly1305.
- Для каждого сообщения генерировать уникальный `nonce/iv` (`esp_random()`), хранить anti-replay окно.

3. Защитить OTA
- Разрешить OTA только в maintenance-режиме/по физическому флагу.
- Проверять подпись прошивки (Ed25519/ECDSA) перед установкой.
- Запретить произвольный URL, использовать фиксированный доверенный origin.

### Этап B (устойчивость и безопасность протокола)

4. Добавить строгую валидацию входа
- Проверять JSON parse error и обязательные поля.
- Ограничить размер body (например, 1-2 KB).
- Возвращать корректные коды ошибок без падения.

5. Убрать VLA/небезопасные буферы
- Использовать заранее ограниченные буферы или `std::vector<uint8_t>` с лимитом.
- Явно контролировать null-terminator после дешифрования.

6. Добавить timeout/reconnect Wi‑Fi
- Не блокировать `setup()` бесконечно.
- Нужен конечный timeout + backoff + watchdog-friendly loop.

### Этап C (качество кода)

7. Выключить debug по умолчанию
- `ASSISTENT_DEBUG` только через флаг сборки для dev.
- Не логировать plaintext команд/аргументов.

8. Исправить API безопасного доступа к переменным
- `ToType` должен проверять `nullptr`.
- Примеры должны проверять `rele != nullptr` перед `rele->ON()/OFF()`.

9. JSON через ArduinoJson вместо ручной конкатенации
- Сериализацию делать через `DynamicJsonDocument`/`StaticJsonDocument`.
- Это уберёт проблемы экранирования и часть багов формата.

## 4) Быстрый список задач для следующего коммита

1. Добавить проверку подписи запроса на `/command`, `/clear`, `/ota`.

3. Убрать статический IV, перейти на уникальный IV per-message.
4. Убрать бесконечный `while` при Wi‑Fi connect, ввести timeout.
